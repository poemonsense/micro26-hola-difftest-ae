\subsection{Abstract}

This artifact contains the HOLA synthesizable reference model (SVM), the
NutShell RISC-V processor used as the design under test, a Linux boot image,
and an automated Artifact Evaluation workflow.  It reproduces the RCache
trade-off data in Figures 11--13, the phased Linux-boot plot in Figure 14, and
the RCore Chisel line counts in Table 4.  Every hardware experiment builds and
runs a Verilator simulator.  FPGA and Palladium platforms are not used.

\subsection{Artifact check-list (meta-information)}

{\small
\begin{itemize}
  \item {\bf Program:} SVM and NutShell, written in Chisel/Scala, SystemVerilog,
        C, C++, Python, and Bash.
  \item {\bf Compilation:} Mill, Chisel, LLVM/Clang or GCC, and Verilator.
  \item {\bf Binary:} The repository includes the NutShell Linux boot image;
        all simulator binaries are built by the workflow.
  \item {\bf Data set:} NutShell \texttt{ready-to-run/linux.bin}.
  \item {\bf Run-time environment:} OpenXiangShan xs-env, x86-64 Ubuntu 24.04
        Docker.
  \item {\bf Hardware:} A 16-core x86-64 host with at least 32 GiB RAM is
        recommended; no accelerator or programmable logic is required.
  \item {\bf Execution:} Serialized emulator builds and parallel Verilator runs.
  \item {\bf Metrics:} Evict miss rate, instruction miss rate, refill bandwidth
        overhead, IPC, load/store IPC, and physical Chisel lines of code.
  \item {\bf Output:} Five Markdown reports, one generated Figure 14 PDF plot,
        and raw logs.
  \item {\bf Experiments:} MICRO 2026 paper Figures 11--14 and Table 4.
  \item {\bf How much disk space required (approximately)?:} 20 GiB, including
        the Docker image, dependency caches, build products, and raw logs.
  \item {\bf How much time is needed to prepare workflow (approximately)?:}
        20--40 minutes, depending on
        network and package-cache speed.
  \item {\bf How much time is needed to complete experiments (approximately)?:}
        2--4 hours on a 16-core host.
  \item {\bf Publicly available?:} The artifact and pinned sources are on
        \href{https://github.com/poemonsense/micro26-hola-difftest-ae}{GitHub};
        the archival release is available at
        \href{https://doi.org/10.5281/zenodo.XXXXXXX}{Zenodo}; an evaluation
        archive supports anonymous or offline access.
  \item {\bf Code licenses (if publicly available)?:} Apache License 2.0 for SVM,
        Mulan PSL v2 for NutShell and difftest, and the license files shipped
        with other nested dependencies.
  \item {\bf Workflow automation framework used?:} Fail-fast Bash scripts plus
        Python reporting.
\end{itemize}
}

\subsection{Description}

\subsubsection{Required commands}

Run only the following three numbered commands, in order, to complete the
evaluation.

\noindent\fbox{\textbf{Command 1}}\quad\textbf{Clone and enter the artifact.}

\begin{verbatim}
git clone --recursive \
  https://github.com/poemonsense/\
micro26-hola-difftest-ae
cd micro26-hola-difftest-ae
\end{verbatim}

\noindent\fbox{\textbf{Command 2}}\quad\textbf{Pull the Docker environment.}

\begin{verbatim}
docker pull \
ghcr.io/openxiangshan/xs-env:ubuntu-24.04
\end{verbatim}

\noindent\fbox{\textbf{Command 3}}\quad\textbf{Run the complete AE workflow.}

\begin{verbatim}
docker run --rm --name hola-ae \
  -v "$PWD:/ae" -w /ae \
ghcr.io/openxiangshan/xs-env:ubuntu-24.04 \
  bash -lc './run-all.sh'
\end{verbatim}

\subsubsection{Command explanation (no action required)}

\paragraph{Command 1.}
Clones the artifact and its pinned NutShell and SVM submodules.

\paragraph{Command 2.}
Downloads the OpenXiangShan xs-env image containing the complete build and
simulation environment.

\paragraph{Command 3.}
Starts the complete evaluation.  The workflow automatically performs the
following stages; the reviewer does not invoke them separately:

\begin{sloppypar}
\begin{enumerate}
  \item \path{setup/setup.sh} checks the environment and initializes the pinned
        source projects.
  \item \path{build/build.sh} builds the Verilator simulators needed by all
        evaluated RCache configurations.
  \item \path{run/run.sh} boots
        \path{NutShell/ready-to-run/linux.bin} on each simulator and records the
        firmware output and simulator snapshots.
  \item \path{report/report.sh} computes the metrics and generates the Figure
        11--14 and Table 4 evaluation files listed in Section A.4.
\end{enumerate}
\end{sloppypar}

\subsection{Evaluation and expected results}

A successful complete run creates the following files:

\begin{verbatim}
evaluation/figure11.md
evaluation/figure12.md
evaluation/figure13.md
evaluation/figure14.md
evaluation/figure14.pdf
evaluation/table4.md
\end{verbatim}

Each Markdown report contains both measured values and the paper values, followed
by a comparison at the paper's displayed precision and the raw-log provenance
and formula for each measured point.  The expected paper data are shown below.
Figures 11--13 use the final hardware-counter values printed by the firmware;
Figure 14 uses periodic simulator-counter snapshots.

\paragraph{Figure 11.}
Evict miss rate is
\[
100\times\mathtt{cache\_evict\_miss}/\mathtt{cache\_evict}.
\]

\begin{center}
\begin{tabular}{lrrr}
\hline
Cache & 4-way & 8-way & 16-way \\
\hline
256 KiB & X & 2.76\% & 3.42\% \\
512 KiB & 0.60\% & 0.41\% & 0.36\% \\
1 MiB   & 0.06\% & 0.01\% & 0.00\% \\
2 MiB   & 0.02\% & 0.00\% & 0.00\% \\
\hline
\end{tabular}
\end{center}

The X is an expected abort because every block becomes dirty and
non-replaceable.  The supplementary PLRU point reported in the paper text is
0.53\% for the 256 KiB/8-way configuration.

\paragraph{Figure 12.}
Instruction miss rate is

\]

\hline
2-bank, 2-port & 18.32\% & 2.08\% & 0.20\% & 0.19\% \\
2-bank, 3-port & 18.18\% & 1.89\% & 0.02\% & 0.00\% \\

\end{center}

\paragraph{Figure 13.}
The disabled and enabled rows use the Figure 12 miss metric.  Bandwidth

\end{tabular}
\end{center}

\paragraph{Figure 14.}
\path{evaluation/figure14.pdf} contains exactly the paper's four displayed
series: 2-port miss rate, 3-port miss rate, IPC, and load/store IPC.  The plot
should cover approximately $300\times2^{16}$ reference cycles and show the
large early 2-port spikes removed by the third port.  The plotting command does
not add the other diagnostic performance-counter series available in
\texttt{plot.py}.

Both Figure 14 runs use a 512 KiB, 8-way cache with refill-on-read-miss.  The
2-port point uses two banks; the 3-port point uses the paper's four-bank,
three-port design.

The Figure 14 plot uses the RTL's 1,024-cycle simulator-counter snapshots.  The
final hardware-counter values printed by the firmware in the same runs are not
inputs to these curves.

\paragraph{Table 4.}
The expected module counts are 253, 61, 52, 126, 87, 154, 49, 25, and 129
lines for MMU, FETCH, AGU, LSU, ARITH, PRIV, TRAP, COMMIT, and RCore,
respectively, totaling 936 physical Chisel source lines.
